# 智能业务中台项目架构设计文档

## 1. 系统概述
本项目旨在构建一个高度可配置的“智能业务中台”，通过低代码配置（模型、提示词、工具、组件）帮助客户将传统业务系统快速 AI 化。最终呈现为“左侧对话、右侧业务面板”的 Copilot 交互形态，实现 AI 分析与业务数据的实时联动、对比与报告生成。

## 2. 技术栈与工程化选型
* **代码仓库管理：** Monorepo (基于 `pnpm workspace` 或 `Turborepo`)，便于前后端类型共享、组件复用及统一构建。
* **前端展示层 (Frontend)：** Vue 3 (Composition API) + Vite + TypeScript。支持本地业务组件库，以及基于 Module Federation（模块联邦）或 Web Components 的远程组件动态加载。
* **业务网关层 (Backend - Node.js)：** Node.js + TypeScript (Express 或 NestJS)。负责前端资源的聚合、静态资源代理、会话管理、工具接口的代理与鉴权。
* **AI 引擎层 (Backend - Python)：** Python + LangGraph/LangChain + FastAPI。负责处理核心的多 Agent 编排、意图识别、大模型交互及上下文状态管理。

## 3. 系统整体架构

系统分为三个核心层级：表现层（Agentic UI）、业务网关层和 AI 引擎层。

### 3.1 表现层 (Agentic UI)
前端整体为一个左右分栏的 Copilot 容器：
* **左侧 (Chat Context)：** 负责与用户的自然语言交互，展示对话流。
* **右侧 (Workspace/Components)：** 动态业务组件渲染区。
* **事件总线 (ActionBus)：** 核心驱动中枢。AI 分析得出的结构化动作指令，通过 WebSocket 传递到前端后，由 ActionBus 解析并分发给右侧对应的 Vue 组件进行状态更新或弹窗展示。
* **远程组件接入器：** 提供统一的 SDK 规范，客户独立打包的远程组件可通过动态 Script 注入或模块联邦的方式注册到中台，并监听 ActionBus 的事件。

### 3.2 业务网关层 (Node.js)
作为前后端和外部业务系统的桥梁：
* **BFF (Backend for Frontend)：** 处理前端的 HTTP/WebSocket 请求。
* **配置中心服务：** 提供可视化后台界面的 API，用于客户配置模型参数、编写/绑定提示词、注册外部业务 API 为“工具 (Tools)”。
* **工具执行器 (Tool Runner)：** 当 AI 决定调用某个业务接口时，由 Node.js 网关代理发起真实的 HTTP 请求至客户原有的业务系统，确保跨域与鉴权安全。

### 3.3 AI 引擎层 (Python)
处理所有大模型相关的复杂逻辑：
* **Agent 编排：** 将复杂的业务拆解为多步 Agent 协作流程。
* **WSCI 上下文工程：**
    * **Write (写入):** 记录多轮对话与右侧组件反馈的实时数据。
    * **Select (选取):** 针对当前对话意图，动态检索相关的业务提示词和工具描述。
    * **Compress (压缩):** 当对话或数据量过大时，对历史上下文和业务报表数据进行摘要降维。
    * **Isolate (隔离):** 将不同业务场景的系统提示词、临时变量和工具集进行沙箱隔离，防止幻觉和上下文污染。
* **报告生成管道：** 在交互末期，汇总 WSCI 上下文中的所有状态，触发专门的 Report Agent 输出最终业务报告。

## 4. Monorepo 目录结构规划

```text
business-ai-middle-platform/
├── package.json
├── pnpm-workspace.yaml
├── apps/
│   ├── web/                 # 前端 Copilot 应用 (Vue3 + Vite)
│   ├── admin-panel/         # 中台管理后台，用于配置提示词、工具等 (Vue3)
│   ├── api-gateway/         # Node.js 业务网关与 WebSocket 服务
│   └── ai-engine/           # Python AI 核心服务 (FastAPI + LangGraph)
├── packages/
│   ├── shared-types/        # TS 类型定义 (前后端复用接口格式、Action 协议)
│   ├── action-bus/          # 独立封装的前端事件总线系统
│   ├── core-components/     # 中台内置的基础业务组件库
│   └── remote-sdk/          # 供客户开发远程组件时使用的 SDK
└── tools/                   # 构建脚本、CLI 工具等
```

## 5. 核心业务流转过程

1.  **场景初始化：** 客户在中台配置好场景 A（绑定了特定的 LLM、提示词模板 X、接口工具 Y，以及远程组件 Z）。
2.  **前端加载：** 用户打开网页，前端根据场景 A 的配置，动态加载远程组件 Z 渲染在右侧，建立 WebSocket 连接。
3.  **用户输入：** 用户在左侧对话框输入自然语言需求。
4.  **请求处理：** 请求通过 Node.js 网关转发至 Python AI 引擎。
5.  **意图与规划 (AI 引擎)：** AI 引擎利用 WSCI 机制提取上下文，决定调用接口工具 Y，并通过大模型生成工具调用的参数。
6.  **接口执行与下发 (Node.js & ActionBus)：** Python 服务将工具指令回传给 Node.js，Node.js 调用实际业务接口。获取结果后，封装成特定的 Action 协议，通过 WebSocket 推送给前端。
7.  **动态渲染：** 前端 ActionBus 捕获事件，驱动右侧远程组件 Z 更新图表或表单数据。
8.  **对比与反馈：** 用户结合右侧实时数据，在左侧继续提出调整意见（循环步骤 3-7）。
9.  **报告输出：** 用户确认无误后，指令 AI 生成报告，Python 端汇总最终状态生成文档返回前端展示。

对于这样一个高度依赖动态配置、多智能体协作以及复杂上下文管理的 AI 中台，数据层的设计需要同时兼顾**“强结构化的业务配置”**和**“半/非结构化的 AI 上下文”**。

以下是为该智能业务中台补充的**数据层架构设计**：

### 1. 核心存储选型建议

为了降低初期多组件部署的运维复杂度，同时满足不同类型数据的存储需求，推荐以下“三件套”组合：

* **PostgreSQL (核心主库)：** 极度推荐。它不仅能完美处理关系型数据，其原生的 JSONB 格式非常适合存储结构多变的提示词配置和工具定义；更重要的是，通过安装 `pgvector` 插件，可以直接将其作为向量数据库使用，省去初期单独部署 Milvus 或 Qdrant 的成本。
* **Redis (内存/缓存库)：** 必不可少。用于 Node.js 端 WebSocket 的 Session 管理、ActionBus 的分布式消息发布/订阅，以及 Python 端高频 Agent 状态的临时缓存。
* **MinIO / S3 兼容存储 (对象存储)：** 用于存放客户打包部署的“远程组件”静态资源（如 UMD/ESM 格式的 JS/CSS 文件）、用户上传的业务基线数据，以及 AI 最终生成的业务报告（PDF/Word/Excel）。

### 2. 数据分类与归属设计

在 Node.js 和 Python 双后端的架构下，数据层需要根据业务边界进行划分，避免跨语言频繁读写同一张表引发的锁和同步问题：

#### A. 业务与配置元数据 (主要由 Node.js 网关维护)
* **存储位置：** PostgreSQL (关系型表)
* **数据内容：**
    * **租户/用户数据：** 账号、权限、Token。
    * **中台配置数据：** 模型提供商配置（API Key）、业务场景定义（Scene）。
    * **工具与组件注册表：** 业务接口 API 的定义（请求头、参数 Schema）、远程组件的元数据（组件名称、版本、MinIO 下载地址）。
* **交互方式：** Node.js 通过 ORM (如 Prisma 或 TypeORM) 进行 CRUD 操作。配置后台直接调用 Node.js 接口更新这些数据。

#### B. 会话与 Agent 状态数据 (主要由 Python 引擎维护)
这里需要深度结合您提到的 **WSCI (Write, Select, Compress, Isolate)** 上下文工程规范来设计存储：
* **存储位置：** PostgreSQL (JSONB 字段) + Redis
* **数据内容：**
    * **Thread/Session ID：** 当前对话的唯一标识。
    * **Write (写入) 日志：** 原始的 User 与 AI 对话历史（Message History）。
    * **Agent State (LangGraph Checkpoint)：** 存储 Agent 执行到哪一步、中间调用的工具结果、以及**Select**和**Compress**处理后的精简上下文。LangGraph 的 Checkpointer 可以直接对接 PostgreSQL 或 Redis。
    * **Isolate (隔离) 沙箱状态：** 针对特定业务场景生成的临时变量或短期记忆。

#### C. 知识与向量数据 (主要由 Python 引擎维护)
* **存储位置：** PostgreSQL (`pgvector` 扩展)
* **数据内容：**
    * **RAG 知识库：** 业务场景下的行业规范、操作手册的向量化切片。
    * **Few-Shot 示例库：** 过去成功执行的业务工具调用参数示例（用于动态选取放入提示词，提高大模型输出 JSON 的稳定性）。

### 3. 数据层在 Monorepo 中的体现

在之前的目录结构基础上，数据层的引入会增加特定的 ORM 模型和迁移脚本管理：

```text
business-ai-middle-platform/
├── apps/
│   ├── api-gateway/         # Node.js 网关
│   │   └── prisma/          # Prisma Schema 文件，定义核心业务表
│   └── ai-engine/           # Python AI 引擎
│       └── models/          # SQLAlchemy Model，定义 LangGraph 状态表和向量表
├── packages/
│   └── shared-types/        # 这里会包含由数据库 Schema 自动生成的 TS 类型
...
```

### 4. 数据流转示例：一次完整的业务请求

1.  **加载配置：** 用户前端进入某个业务场景。Node.js 从 **PostgreSQL** 读取该场景绑定的组件 URL，从 **MinIO** 获取组件包下发给前端；同时从数据库加载绑定的提示词模板和工具列表，准备转发给 Python。
2.  **执行编排：** Python 收到执行请求，在 **Redis** 中初始化当前执行状态。利用 `pgvector` 从 **PostgreSQL** 检索最匹配的 Few-Shot 示例。
3.  **持久化：** 步骤执行完毕后，Python 触发 WSCI 的压缩机制，将精简后的上下文和执行结果写入 **PostgreSQL** 的 Agent State 表中，以便随时打断或恢复。报告生成后，文件推送到 **MinIO**，并将文件 URL 存入数据库。
